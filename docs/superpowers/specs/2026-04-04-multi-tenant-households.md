# Multi-Tenant Architecture with Household Linking — Design Spec

**Date:** 2026-04-04
**Status:** DEFERRED — implement after backup/restore and Thunderbird discovery

---

## Goals

1. Multiple users can register accounts on a single Expensor instance, each with fully isolated data, readers, and configuration.
2. Users can form a **Household** — a named group whose members can share a combined transaction view across accounts.
3. OAuth tokens and `client_secret` files are stored in the database with application-level encryption, not as plaintext files on the filesystem.
4. The migration path for existing single-user installations is non-destructive and zero-downtime.

---

## 1. Authentication

### 1.1 User accounts

```sql
CREATE TABLE users (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email         TEXT NOT NULL UNIQUE,
    password_hash TEXT NOT NULL,          -- bcrypt, cost 12
    display_name  TEXT NOT NULL,
    role          TEXT NOT NULL DEFAULT 'member', -- 'admin' | 'member'
    created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

The **first registered user** is automatically assigned `role = 'admin'`. All subsequent registrations require an invitation (see §1.3) unless `EXPENSOR_OPEN_REGISTRATION=true`.

### 1.2 Sessions (JWT)

Short-lived JWTs (1-hour expiry). No refresh tokens — self-hosted users re-authenticate when the token expires. Claims: `{ user_id, email, role, exp }`.

Token is sent in `Authorization: Bearer <token>`. All `/api/*` routes except `/api/auth/*` and `GET /api/health` require a valid token. Middleware injects `user_id` into the request context.

Sessions are not stored in the DB (stateless JWT). If a forced logout / token revocation is needed later, a `user_sessions` denylist table can be added.

### 1.3 Instance registration policy

| Situation | Behaviour |
|-----------|-----------|
| No users exist | Registration open to anyone; first registrant becomes admin |
| Users exist, `EXPENSOR_OPEN_REGISTRATION=false` (default) | Registration requires an invite token generated by admin |
| Users exist, `EXPENSOR_OPEN_REGISTRATION=true` | Any email can register |

Admin invite flow: `POST /api/admin/invites` generates a single-use token with a 7-day expiry. Recipient visits `/setup/register?token=<invite>`.

### 1.4 API endpoints (auth)

```
POST /api/auth/register        { email, password, display_name, invite_token? }
POST /api/auth/login           { email, password } → { token, user }
GET  /api/auth/me              → { id, email, display_name, role }
PUT  /api/auth/me              { display_name?, password? }
POST /api/admin/invites        → { invite_token, expires_at }  [admin only]
```

## 2. Data isolation

Every user-owned table gains a `user_id UUID NOT NULL REFERENCES users(id)` column. All existing queries gain a `WHERE user_id = $user_id` clause enforced by the Go data layer (not just by convention).

### Affected tables

| Table | Change |
|-------|--------|
| `transactions` | Add `user_id` |
| `transaction_labels` | Inherited via FK to `transactions` |
| `labels` | Add `user_id` |
| `categories` | Add `user_id` |
| `buckets` | Add `user_id` |
| `rules` (user-source) | Add `user_id`; system rules remain global (no `user_id`) |
| `app_config` | Add `user_id`; each user has their own settings |

The `Storer` interface gains `userID string` on every method, propagated from the request context. The HTTP middleware injects the authenticated user ID; no handler manually looks it up.

### 2.1 System rules

System rules (seeded from `rules.json`, `source = 'system'`) have `user_id = NULL` and are visible to all users. User-created rules have the owning `user_id`. `MergeRules` now merges global system rules with the current user's rules.

## 3. Credential & state storage

### 3.1 The problem with plaintext files

Current layout:
```
data/
  client_secret_gmail.json   ← OAuth app credentials (plaintext)
  token_gmail.json           ← user's OAuth access+refresh token (plaintext, highly sensitive)
  state.json                 ← processed message IDs
  active_reader              ← which reader is configured
```

Risks:
- Any process or user with read access to `data/` can extract live OAuth tokens
- Files are trivially included in accidental backups, docker volume leaks, or `tar` archives
- No per-user separation

### 3.2 Move all credentials to the database with encryption

A new `user_credentials` table stores every credential type, encrypted at rest:

```sql
CREATE TABLE user_credentials (
    id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id          UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    reader_name      TEXT NOT NULL,   -- 'gmail', 'thunderbird'
    credential_type  TEXT NOT NULL,   -- 'client_secret' | 'oauth_token' | 'reader_config'
    encrypted_value  BYTEA NOT NULL,  -- AES-256-GCM ciphertext
    iv               BYTEA NOT NULL,  -- 12-byte GCM nonce (unique per row)
    created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (user_id, reader_name, credential_type)
);
```

### 3.3 Encryption key (`EXPENSOR_SECRET_KEY`)

A 32-byte random key provided at deployment time:

```bash
# Generate once, store permanently in your deployment configuration:
openssl rand -hex 32
```

Set as `EXPENSOR_SECRET_KEY=<hex>` in the environment. Used for AES-256-GCM encryption of all credential values.

**If not set:** Expensor auto-generates a key on first start and writes it to `data/.secret_key` (mode `0600`, owned by the `expensor` user) with a startup warning. This is acceptable for a single-machine deployment where `data/` is already only accessible to the service user. Production deployments should set the env var and not rely on the auto-generated file.

**Key rotation:** Not in scope for v1. When needed: re-encrypt all `user_credentials` rows with the new key, then update the env var.

### 3.4 Encryption algorithm

AES-256-GCM. Each row gets a unique 12-byte random IV. The `user_id + reader_name + credential_type` triple is used as associated data (AAD) to bind the ciphertext to its owner — decrypting a token as a different user or for a different reader produces an authentication error.

```
ciphertext = AES-256-GCM-Encrypt(key=SECRET_KEY, iv=random_iv, plaintext=JSON, aad="user_id:reader_name:type")
```

### 3.5 State → database

`state.json` moves to a `user_state` table. This eliminates per-user state file proliferation, makes state part of regular DB backups, and enables ACID semantics on state writes:

```sql
CREATE TABLE user_state (
    user_id       UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    reader_name   TEXT NOT NULL,
    message_key   TEXT NOT NULL,       -- SHA-256 hash (same as state.json key)
    processed_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (user_id, reader_name, message_key)
);
```

The `state.Manager` interface gains a `userID` parameter. For single-user compatibility during migration, existing `state.json` entries are imported into `user_state` on first startup.

### 3.6 Active reader → app_config

`data/active_reader` becomes `app_config` key `active_reader` per user. Handled by the existing `SetAppConfig`/`GetAppConfig` infrastructure.

### 3.7 What remains on the filesystem

After this change, `data/` contains only:
- `.secret_key` (if auto-generated; mode `0600`)
- Nothing else by default

The `VOLUME ["/app/data"]` in Dockerfile is retained for the secret key file, but its security posture improves dramatically — no OAuth tokens or credentials live there.

## 4. Household architecture

### 4.1 Data model

```sql
CREATE TABLE households (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name        TEXT NOT NULL,
    created_by  UUID NOT NULL REFERENCES users(id),
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE household_members (
    household_id        UUID NOT NULL REFERENCES households(id) ON DELETE CASCADE,
    user_id             UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    role                TEXT NOT NULL DEFAULT 'member',  -- 'owner' | 'member'
    share_transactions  BOOLEAN NOT NULL DEFAULT true,   -- opt-in to household view
    joined_at           TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (household_id, user_id)
);
```

A user can belong to **at most one household**. Household `owner` can invite members and rename the household; members can only change their own `share_transactions` preference.

### 4.2 Household transaction view

`GET /api/households/{id}/transactions` returns the union of transactions from all members where `share_transactions = true`, with each row annotated with `owner: { display_name }`. Supports the same filters (date, category, label, etc.) as the individual transaction endpoint.

Members can see household transactions but cannot edit each other's transactions, labels, rules, or configuration. The household view is read-only for non-owners of each transaction.

### 4.3 Shared vs private resources

| Resource | Shared in household? |
|----------|---------------------|
| Transactions | Yes, if `share_transactions = true` (per-member toggle) |
| Labels | No — each user manages their own |
| Rules | No — each user manages their own |
| Categories / Buckets | No — each user manages their own |
| App config | No — per-user settings |

Future: a household may define shared labels/categories (e.g. "Family Holiday 2026") as an opt-in feature.

### 4.4 API endpoints (household)

```
POST   /api/households                           Create household [any authed user]
GET    /api/households/{id}                      Get household details + members
PUT    /api/households/{id}                      Rename [owner only]
DELETE /api/households/{id}                      Disband [owner only]
POST   /api/households/{id}/members/invite       Invite by email [owner only]
DELETE /api/households/{id}/members/{user_id}    Remove member [owner, or self-remove]
PUT    /api/households/{id}/members/me           Update share_transactions toggle
GET    /api/households/{id}/transactions         Combined transaction view
```

## 5. Migration from single-user

On first startup after this upgrade:

1. Detect there are no rows in the `users` table.
2. Create a default admin user:
   - Email: `admin@localhost`
   - Password: randomly generated, printed **once** to stdout/logs in plaintext: `[EXPENSOR] First-run admin password: <password>` — this is the only time it appears; the user should change it immediately.
3. Migrate all existing data (`transactions`, `labels`, `categories`, `buckets`, `rules`, `app_config`) to `user_id = <admin user id>`.
4. Import `data/state.json` into `user_state` under the admin user, then delete the file.
5. Import `data/client_secret_*.json` and `data/token_*.json` into `user_credentials` (encrypted), then delete the plaintext files.
6. Move `data/active_reader` to `app_config[active_reader]`, then delete the file.

If any step fails, the migration is rolled back and the old files are untouched. A `--dry-run` flag allows previewing the migration without executing it.

## 6. Security model summary

| Threat | Mitigation |
|--------|-----------|
| Plaintext OAuth token on disk | Tokens stored in DB, AES-256-GCM encrypted |
| DB dump leaks tokens | Encrypted values useless without `EXPENSOR_SECRET_KEY` |
| Cross-user token decryption | AAD binds ciphertext to `user_id:reader_name:type` |
| Weak auto-generated key | Logged warning; production env var recommended |
| Household member sees another user's edits | Household view is read-only; write APIs enforce user ownership |
| Unauthenticated API access | JWT middleware on all non-auth routes |
| Admin token theft | Short JWT expiry (1 hour); no persistent refresh tokens |

---

## 7. Impact on backup/restore

The backup spec (v1) stores `state.json` as a top-level `state` object. After this migration, the backup spec becomes:
- `state` object is removed (state is now in the DB as `user_state`)
- `user_credentials` are **excluded** from backups (they are encrypted and machine/instance-tied — re-uploading credentials is part of the onboarding flow on a new installation)
- All other tables are backed up per-user; restoring a backup restores that user's data

`backup_version` increments to **2** when multi-tenant ships.

---

## 8. Forward-compatibility for new features

Every feature built after multi-tenant must:
1. Add `user_id` to any new user-data table
2. Filter all queries by the authenticated user ID from context
3. Ensure new credential types are stored in `user_credentials` (never as files)
4. Include new tables in the backup's `backup_version` increment
