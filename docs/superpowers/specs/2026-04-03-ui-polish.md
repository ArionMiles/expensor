## 1. Date Range Picker (#7)

### Goal
Replace the two separate `<input type="date">` fields in the Transactions filter bar with a single cohesive date-range picker that matches the app's dark/light aesthetic.

### Library
`react-day-picker` v8 (MIT licence, headless, fully styleable via CSS modules or Tailwind utility classes). No additional dependencies beyond `react-day-picker` and its peer `date-fns`.

### Component: `DateRangePicker`

**File:** `frontend/src/components/DateRangePicker.tsx`

**Behaviour:**
- Renders a button labelled "All time" when no range is selected; "Jan 15 – Mar 3, 2025" when active.
- Clicking the button opens a floating popover containing a two-month calendar.
- User selects start date then end date; partial selection (start only) is valid until end is chosen.
- Clear button inside the popover resets to "All time".
- Popover closes on outside click or Escape.
- Controlled component: accepts `value: { from?: Date; to?: Date }` and `onChange` callback.

**Styling:**
- Popover uses the existing `--card` / `--border` / `--primary` CSS variables so it automatically respects the theme toggle.
- Selected day uses `--primary` background; range fill uses `--accent`.
- No custom CSS file — all styles applied via Tailwind class overrides on `react-day-picker`'s class names.

### Integration
`Transactions.tsx` currently computes `from` / `to` filter state independently. Replace those two state variables and their `<input>` elements with a single `<DateRangePicker>` binding. The existing `?from=` / `?to=` query params sent to `GET /api/transactions` are unchanged.

### Backend
No changes required.

## 2. Time Display (#8)

### Goal
Show transaction time (HH:MM) alongside the date in the Transactions table. The `timestamp` field already stores full precision in the database; `formatDate` currently discards the time component.

### Change

**File:** `frontend/src/lib/utils.ts`

Update `formatDate` to accept an optional `includeTime` flag:

```ts
export function formatDate(isoString: string, includeTime = false): string {
  const date = new Date(isoString)
  const opts: Intl.DateTimeFormatOptions = {
    day: '2-digit',
    month: 'short',
    year: 'numeric',
    ...(includeTime ? { hour: '2-digit', minute: '2-digit', hour12: false } : {}),
  }
  return new Intl.DateTimeFormat('en-IN', opts).format(date)
}
```

Output examples:
- Without time: `15 Jan 2025`
- With time: `15 Jan 2025, 14:32`

**File:** `frontend/src/pages/Transactions.tsx`

Pass `includeTime={true}` to the date cell renderer. The timestamp column is currently rendered as `formatDate(tx.timestamp)` — change to `formatDate(tx.timestamp, true)`.

### Backend
No changes required.

## 3. Column Sort (#9)

### Goal
Allow the user to sort transactions by date ascending or descending by clicking the Date column header.

### Backend

**File:** `backend/internal/store/store.go`

Add two fields to `ListFilter`:
```go
SortBy  string // "timestamp" (only supported value for now)
SortDir string // "asc" | "desc"; default "desc"
```

Update `ListTransactions` to use these in the `ORDER BY` clause:
```go
// Validated/sanitised before use — only "timestamp" accepted for SortBy.
orderClause := "ORDER BY t.timestamp DESC"
if f.SortBy == "timestamp" {
    dir := "DESC"
    if strings.ToLower(f.SortDir) == "asc" {
        dir = "ASC"
    }
    orderClause = fmt.Sprintf("ORDER BY t.timestamp %s", dir)
}
```

`SortBy` is validated against an allowlist (`{"timestamp"}`) before use in SQL to prevent injection.

**File:** `backend/internal/api/handlers.go`

Parse new query params in `HandleListTransactions`:
```go
f.SortBy  = r.URL.Query().Get("sort_by")   // default ""
f.SortDir = r.URL.Query().Get("sort_dir")  // default ""
```

### Frontend

**File:** `frontend/src/pages/Transactions.tsx`

- Date column header becomes a `<button>` showing "Date" + an arrow icon (↓ default, ↑ when ASC).
- Clicking toggles: `desc` → `asc` → `desc`.
- `sort_by=timestamp&sort_dir=asc` appended to the transactions query.
- Active sort direction stored in component state alongside existing filter state.

### API contract
`GET /api/transactions?sort_by=timestamp&sort_dir=asc`

## 4. Dual Theme (#13)

### Goal
Full dark + light mode support with a three-way toggle (dark / light / system). System mode respects `prefers-color-scheme`. Preference persisted in `localStorage`.

### Architecture

**File:** `frontend/src/components/ThemeProvider.tsx` (new)

```ts
type Theme = 'dark' | 'light' | 'system'

const ThemeContext = createContext<{
  theme: Theme
  setTheme: (t: Theme) => void
}>({ theme: 'system', setTheme: () => {} })

export function ThemeProvider({ children }: { children: ReactNode }) {
  const [theme, setThemeState] = useState<Theme>(
    () => (localStorage.getItem('theme') as Theme) ?? 'system'
  )

  useEffect(() => {
    const root = document.documentElement
    const resolved =
      theme === 'system'
        ? window.matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light'
        : theme
    root.classList.remove('dark', 'light')
    root.classList.add(resolved)
    localStorage.setItem('theme', theme)
  }, [theme])

  // Also react to OS-level changes when theme === 'system'
  useEffect(() => {
    if (theme !== 'system') return
    const mq = window.matchMedia('(prefers-color-scheme: dark)')
    const handler = () => {
      document.documentElement.classList.remove('dark', 'light')
      document.documentElement.classList.add(mq.matches ? 'dark' : 'light')
    }
    mq.addEventListener('change', handler)
    return () => mq.removeEventListener('change', handler)
  }, [theme])

  const setTheme = (t: Theme) => setThemeState(t)
  return <ThemeContext.Provider value={{ theme, setTheme }}>{children}</ThemeContext.Provider>
}

export const useTheme = () => useContext(ThemeContext)
```

**File:** `frontend/src/main.tsx`

Wrap `<App>` with `<ThemeProvider>`.
